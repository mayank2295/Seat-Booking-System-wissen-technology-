1. Project Overview

The Seat Booking System is a real-time web application designed for corporate environments where the number of employees exceeds the available office seats. It manages seat allocation fairly through an automated weekly batch rotation mechanism, live seat booking, leave management, and an admin panel.

2. Problem Statement

An office has 80 employees but only 50 physical seats. Without a system:

  • Employees arrive to find no seats available
  • Manual coordination via emails/spreadsheets is error-prone
  • No visibility into who is coming or who is on leave
  • No fairness — same people occupy the best seats every day
  • When someone takes leave, their seat goes wasted for the entire day

3. Solution Approach

The system solves this through four key mechanisms:

  1. Weekly Batch Rotation: 80 employees are split into 2 batches of 40. Each batch gets priority access to regular seats on alternating days, rotating every week.
  2. Floater Seats: 10 seats (S-41 to S-50) are permanently open to everyone, allowing the non-priority batch to still come in when needed.
  3. Leave-to-Floater Conversion: When an employee declares leave, their booked seat is automatically released and becomes available as a temporary floater for others.
  4. Real-Time Dashboard: All bookings and releases are broadcast instantly to every connected user via Server-Sent Events (SSE).

Three-tier architecture:

7. Core Business Logic

7.1 Batch Rotation Algorithm

The rotation algorithm determines which batch has priority on any given date without any manual intervention:

// Reference date: a Monday that marks the start of Week 1
const WEEK_REF_DATE = "2026-01-05";  // Set in .env

function getWeekNumber(date) {
    daysElapsed  = (date - referenceDate) / (24 * 60 * 60 * 1000)
    weeksElapsed = floor(daysElapsed / 7)
    weekParity   = weeksElapsed % 2    // 0 → Week 1, 1 → Week 2
    return weekParity + 1              // Returns 1 or 2
}

Rotation Schedule:

WeekBatch 1Batch 2
Week 1 (odd)Mon, Tue, WedThu, Fri
Week 2 (even)Thu, FriMon, Tue, Wed

The isTeamDay(batch, date) function checks if today is a "team day" for a given batch by matching the day of the week against the schedule for the current rotation week.

7.2 Seat Classification

All 50 seats are stored in one table. The type is determined by ID:

  • Seats 1–40 (Regular): Only bookable by the priority batch on their assigned days.
  • Seats 41–50 (Floater): Open to everyone, any day (subject to 3 PM cutoff for non-priority batch).
  • Temporary Floaters: Regular seats released via cancellation or leave. These appear with a distinct orange color on the dashboard.
seatType = (seat.id <= 40) ? "regular" : "floater"
tempFloater = (seat was cancelled/released for this date)

7.3 Booking Validation Chain

Every booking request passes through 7 sequential validations:

POST /api/book { employeeId, seatId, date }

1. isWeekday(date)?          → No weekends allowed
2. userExists(employeeId)?   → Must be valid employee
3. seatExists(seatId)?       → Must be valid seat
4. alreadyBooked(user,date)? → One seat per person per day
5. onLeave(user, date)?      → Cannot book if on leave
6. seatTaken(seat, date)?    → No double-booking same seat
7. batchPermission?          →
     If team day     → any seat allowed
     If not team day → floater only + after 3PM cutoff

If any check fails, the booking is rejected with a specific error message. This chain prevents all edge cases including race conditions.

7.4 Leave-to-Floater Conversion

This is the key differentiating feature. When an employee declares leave:

DECLARE LEAVE (employeeId, date):
   1. Validate: Is it a weekday?
   2. Validate: Is it the user's team day?
   3. Check: Does user have an active booking for this date?
   4. If YES → Cancel booking (status = 'cancelled')
              → The freed seat becomes a TEMPORARY FLOATER
              → Broadcast SSE event to all connected clients
   5. Insert leave record into leaves table
   6. Return: "Leave declared. Seat S-XX now available as floater."
Key Insight: Without this feature, when a priority-batch member takes leave, their seat sits empty all day and no one else can use it. With this feature, the seat automatically becomes available to anyone — increasing seat utilization from ~80% to ~95%.

Floater count formula:

Available Floaters = 10 (permanent) + released_seats + leave_freed_seats

7.5 3 PM Cutoff Rule

Non-priority batch employees can only book floater seats after 3:00 PM the day before the booking date. This gives the priority batch the entire working day to make their bookings first.

cutoffTime = bookingDate - 1 day, 15:00:00
if (currentTime < cutoffTime) → REJECT "Floaters available after 3 PM"

8. Real-Time Updates (SSE)

Why Real-Time?

Without real-time updates, User A books seat S-05 but User B's screen still shows it as available. User B clicks it and gets an error. Poor user experience.

How SSE Works

1. Browser opens: GET /api/stream (persistent HTTP connection)
2. Server stores this connection in a Set: sseClients.add(res)
3. Connection stays open indefinitely

4. When ANY user books/releases/declares leave:
   → Server calls broadcastSSE(event, data)
   → Loops through ALL clients in the Set
   → Sends the event to each connected browser

5. Browser receives event → auto-refreshes seat grid + activity log

6. If connection drops → browser auto-reconnects after 3 seconds
7. On logout → connection is cleanly closed, reconnection flag disabled

Why SSE over WebSocket?

FactorSSEWebSocket
DirectionOne-way (server → client)Two-way
ComplexitySimple HTTPHandshake protocol
Library neededNone (native browser API)Usually needs socket.io
Auto-reconnectBuilt-inMust implement manually
Our need✅ SufficientOverkill

9. Authentication & Security

Password Handling

  • Passwords are hashed using bcrypt with 10 salt rounds before storing in the database.
  • On login, bcrypt.compare(input, storedHash) verifies the password without ever decrypting it.
  • Even if the database is compromised, passwords cannot be reversed.

Dual Session Strategy

Session TypeStorageUsed For
Server SessionMySQL (express-mysql-session)Admin EJS pages, persists across server restarts
Client SessionBrowser sessionStorageSPA navigation, client-side state

Security Measures

  • Helmet.js: Sets HTTP security headers (XSS protection, content-type sniffing, etc.)
  • CSRF Protection: Admin panel requires active server session.
  • Role-based Access: Middleware checks req.session.user.role === 'admin' before allowing access to admin routes.
  • Input Validation: All API endpoints validate required fields before processing.

11. Frontend — SPA Architecture

Single Page Application Design

The entire user-facing application is a single index.html file with three views:

  1. Landing View: Hero section, feature cards, animated counters, call-to-action. Uses IntersectionObserver for scroll-reveal animations.
  2. Login View: Split-layout design with email/password form. Error feedback with animated banner.
  3. Dashboard View: Date picker, info cards (total/available/booked/batch/leave), booking banner, seat grids (regular + floater), legend, leave management, and real-time activity log.

View Switching Mechanism

// Only one view is visible at a time via CSS class
.view { display: none; }
.view.active { display: block; animation: fadeIn; }

// JavaScript toggles the 'active' class
function navigate(viewName) {
    allViews.forEach(v => v.classList.remove('active'));
    targetView.classList.add('active');
}

Design System

  • Apple-inspired minimal aesthetic with Inter font
  • CSS custom properties for consistent theming
  • Glass morphism effects on navigation
  • Smooth page transitions with overlay animation
  • Confetti animation on successful booking
  • Fully responsive — works on mobile, tablet, and desktop

14. Setup & Deployment

Prerequisites

  • Node.js v16 or higher
  • MySQL 8 or higher

Installation Steps

  1. Clone: git clone https://github.com/mayank2295/Seat-Booking-System-wissen-technology-.git
  2. Install: npm install
  3. Setup database: Run schema.sql in MySQL
  4. Create .env file: 
    DB_HOST=localhost
DB_USER=root
DB_PASSWORD=your_password
DB_NAME=seat_booking
DB_PORT=3306
SESSION_SECRET=your_secret_key
TOTAL_SEATS=50
WEEK_REF_DATE=2026-01-05
PORT=3000
  5. Start: npm start
  6. Open: http://localhost:3000
